BiG Decimal?

Is that what you want?

Context

  • Financial software full of BigDecimal
  • Need for accuracy
  • Need to have a big cache of prices
  • "Only" 8GB of memory

Primitives in Scala

  • AnyVal extends Any with NotNull
  • Null not allowed, optimization allowed
  • Array[Double] <=> double[]

On the bench

    val ITER: Int = 10000000
    val myBds: Array[BigDecimal] =
      for (i <- Array.range(1, ITER))
        yield BigDecimal(Random.nextDouble, MathContext.DECIMAL32)
    val myDoubles: Array[Double]  = myBds.map(_.toDouble)

    val bdSize = ObjectSizeCalculator.getObjectSize(myBds)
    val doubleSize = ObjectSizeCalculator.getObjectSize(myDoubles)

    println("BigDecimal size : " + humanReadableByteCount(bdSize))
    println("Double size     : " + humanReadableByteCount(doubleSize))
    println("Ratios : " + bdSize.toDouble / doubleSize.toDouble)

    val ts1 = System.nanoTime()
    myBds.map(_ * 1000)
    val ts2 = System.nanoTime()
    println("Time to process BigDecimal : "+humanReadableduration(ts1, ts2))

    val ts3 = System.nanoTime()
    myDoubles.map(_ * 1000)
    val ts4 = System.nanoTime()
    println("Time to process Double     : "+humanReadableduration(ts3, ts4))
    println("Ratios : " + (ts2-ts1).toDouble / (ts4-ts3).toDouble)

Characteristics of BigDecimal

  • Very Accurate if well used
  • Slower than primitives
  • Memory eager
  • Need a minimum of knowledge
  • Not a standard

Examples

A simple mistake
BigDecimal(123456789, new MathContext(5)) / BigDecimal(10) //1.2346E+7
A scala weirdness
BigDecimal(123456789, new MathContext(5)) // 123456789
BigDecimal(123456789l, new MathContext(5)) // 1.2346E+8
Differences on precison
1d / 3d //0.3333333333333333
BigDecimal(1) / BigDecimal(3) //0.3333333333333333333333333333333333
new java.math.BigDecimal(1).divide(new java.math.BigDecimal(3)) //java.lang.ArithmeticException

Long as an option?

  • Efficient
  • Convenient for fixed point numbers
  • Very limited
  • Overhead in the code

Double as an option?

  • Fixed accuracy, 15 digits
  • Processor optimized
  • 64bits only
  • Binary, not exact decimal value
  • Rounding needed
  • Simple

Conclusion

If you don't need more than 15 digits
Use Double!
